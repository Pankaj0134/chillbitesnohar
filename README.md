# Chill Bites Nohar — Loyalty Website

A premium, mobile-first React site for QR-based loyalty rewards, customer
accounts, Google Reviews, and Instagram growth.

## Run it

```bash
npm install
npm run dev       # local dev server
npm run build     # production build → dist/
```

**Before the loyalty card will work, you need a Supabase project.**
See `SUPABASE_SETUP.md` for complete from-scratch setup steps (~15 min).
Without it, the site still renders, but sign-in/sign-up will fail with a
console error pointing you to that guide.

## How accounts work

- Customers create an account with **phone number + password** — no OTP,
  no SMS cost. This was a deliberate tradeoff: it's genuinely free, but
  means there's no proof a phone number actually belongs to the person
  typing it in. Fine for a loyalty app; see `SUPABASE_SETUP.md` for the
  full reasoning.
- Once signed in, loyalty progress lives in Supabase (table:
  `loyalty_accounts`), not localStorage — it follows the customer across
  devices.
- **Claiming a token requires a fresh QR code generated by staff at the
  counter** (table: `claim_codes`). This is the actual anti-fraud
  mechanism — without it, a customer could just open the site from home
  and tap "claim" repeatedly. The flow:
  1. Staff opens `/staff`, enters their PIN, clicks **Generate QR Code**.
  2. A 6-digit code appears as a QR, valid for **5 minutes**, usable
     **once**.
  3. The customer scans it with their phone's camera — this opens the
     site with the code in the URL (`?claim=482917`), which is read once
     and then stripped from the visible address bar.
  4. Clicking **Claim Today's Token** calls `claim_token(code)`, which
     checks server-side that the code exists, hasn't expired, and hasn't
     been used — all inside the Postgres function, not the browser, so
     it can't be bypassed by replaying an old URL or editing client code.
  5. The 12-hour cooldown and 4-token cap are still enforced on top of
     the code check, in the same function.
- Existing visitors are **not** migrated from any old localStorage
  progress; every new account starts at 0 tokens, per your instruction.

### What this does and doesn't protect against

- **Does stop:** claiming from home, reusing an old QR/link, claiming
  more than once per generated code, claiming faster than the cooldown.
- **Doesn't stop:** a customer photographing the counter screen and
  handing their phone to a friend standing next to them within the same
  5 minutes (the code doesn't check *who* scans it, only *that* it's
  fresh and unused) — a low-stakes edge case for a free-fries-tier
  reward, not worth solving here.

## Staff / owner panel

Visit `/staff` on the deployed site (e.g. `chillbitesnohar.com/staff`).
This route is **not linked anywhere** in the public nav, footer, or any
visible UI — it's meant to live as a bookmark on one trusted device at the
counter (laptop, tablet, or phone — any browser works).

- **Generate QR Code** creates a fresh, 5-minute, one-time claim code and
  displays it as a QR. Show this screen to the customer so they can scan
  it with their phone's camera to claim a token. A live countdown shows
  staff how much time is left.
- **Auto-refresh** (on by default): once the first code is generated, a
  new one appears by itself the instant the previous one expires — no
  need to click again. Leave the panel open on the counter device and
  it keeps working all day. Turn it off if you'd rather control timing
  manually.
- Staff can also enter a **PIN** plus a customer's phone number under
  **Look Up Customer** to see their token count.
- If the customer has 4/4 tokens, staff can mark the reward **redeemed**,
  which resets their count to 0.
- PIN verification happens entirely server-side, inside Postgres functions
  (`staff_lookup`, `staff_redeem`, `generate_claim_code`) — the PIN is
  never checked in the browser, and a wrong PIN and a not-found phone
  number return the same generic error, so the panel doesn't leak which
  one happened.
- You can add more staff PINs anytime via the SQL Editor — see
  `SUPABASE_SETUP.md` step 4 for the exact command.

## Before you launch — replace these placeholders

All in `src/data/content.js`:

- `googleReviewUrl` — your real Google Business review link (`g.page/r/...`)
- `instagramUrl` / `instagramHandle` — your real Instagram profile
- `whatsappUrl` — your WhatsApp number, format `https://wa.me/91XXXXXXXXXX`
- `phoneNumber`, `address`, `hours`
- `mapsEmbedUrl` — get this from Google Maps → Share → Embed a map
- `couponCode` — the real code your staff will honor at checkout
- `GALLERY` image URLs — swap the Unsplash placeholders for real cafe photos

## Stack

React 19 + Vite + Tailwind CSS v4 + Framer Motion + Supabase (auth + database).

## Structure

```
src/
  components/   one file per section (Hero, LoyaltyCard, SignInPanel, StaffPanel, etc.)
  data/content.js   all editable copy, links, and content lives here
  hooks/useAuth.jsx     phone+password auth, session, and account state
  hooks/useLoyalty.js   reflects loyalty_accounts state; claim logic lives server-side
  lib/supabaseClient.js   Supabase client init (reads VITE_SUPABASE_* env vars)
supabase/
  schema.sql                  tables, RLS policies, staff lookup/redeem functions
  claim_token_function.sql    server-side claim logic (cooldown + cap enforcement)
  claim_codes_migration.sql   rotating QR claim codes (the actual anti-fraud layer)
```

## What's intentionally not built

- Password reset UI (see `SUPABASE_SETUP.md` for the manual workaround)
- CAPTCHA / sign-up abuse protection beyond Supabase's defaults
- Admin dashboard for managing rewards copy or viewing all customers at once
  (only single-phone-number lookup exists, by design — see original scope)
